\section{Introdução}
% -------------------------------------------------%

\begin{frame}{Introdução}
    \begin{itemize}
        \item O Markupp é um servidor \textit{open source} e \textit{self-hosted} de anotações em \textit{Markdown}, com interface aberta a humanos e a agentes de IA.
        \item Nasce como atividade de extensão acadêmica do IFSC, em parceria com Sebrae e Startup Garage.
        \item Tem como referência conceitual a prática de \emph{segundo cérebro} digital, consolidada em \emph{Personal Knowledge Management} \cite{forte2022}.
    \end{itemize}

    \vspace{0.5em}
    \textbf{Pilares}
    \begin{itemize}\small
        \item Soberania de dados: o acervo permanece sob controle direto do usuário.
        \item Interface aberta a múltiplos clientes, sem capturar o usuário em um aplicativo único.
        \item \textit{Open source}, \textit{self-hosted} e com licença permissiva.
    \end{itemize}
\end{frame}

% -------------------------------------------------%
\section{Tamanho do mercado}
% -------------------------------------------------%

\begin{frame}{Dimensão do problema e alcance potencial}
    \begin{itemize}
        \item A prática de construir um \emph{segundo cérebro} se firmou como referência cultural do \emph{Personal Knowledge Management} na última década \cite{forte2022}.
        \item O \textit{Markdown} se consolidou como formato padrão de anotação em comunidades técnicas e acadêmicas \cite{commonmark}.
        \item Cresce o volume de saídas geradas por agentes de IA que precisam ser armazenadas e organizadas de forma estruturada.
        \item O Notion ultrapassou 100 milhões de usuários \cite{notion100m}; o Obsidian, que não divulga números oficiais, estima cerca de um milhão \cite{obsidiancult}.
        \item Esse público de ferramentas proprietárias é um proxy de quem se beneficiaria de uma alternativa aberta.
    \end{itemize}
\end{frame}

% -------------------------------------------------%
\section{Problema}
% -------------------------------------------------%

\begin{frame}{Problema}
    \begin{itemize}
        \item Dependência de plataformas proprietárias, com o acervo hospedado na nuvem do fornecedor e fora do controle do usuário.
        \item Fragmentação entre clientes: o conteúdo fica amarrado a um aplicativo único, sem interface aberta a outros usos.
        \item Ausência de infraestrutura aberta pronta para integração com agentes de IA sobre a mesma base de dados.
        \item Lacuna em desempenho e escalabilidade de soluções que acompanhem o crescimento da base sob \textit{self-hosting}.
    \end{itemize}
\end{frame}

% -------------------------------------------------%
\section{Solução}
% -------------------------------------------------%

\begin{frame}{Solução}
    \textbf{O que o Markupp entrega}
    \begin{itemize}\small
        \item Servidor \textit{self-hosted} com CRUD completo de notas em \textit{Markdown}.
        \item Plugin do Obsidian como cliente humano de referência.
        \item API REST como única interface de acesso ao servidor.
    \end{itemize}

    \vspace{0.5em}
    \textbf{Posicionamento}
    \begin{itemize}\small
        \item \textit{Self-hosted} real: o usuário mantém o acervo em servidor próprio.
        \item API aberta pensada para humanos e agentes desde o projeto.
        \item Licença permissiva e código auditável.
    \end{itemize}
\end{frame}

\begin{frame}{Solução: arquitetura}
    \begin{columns}[T]
        \begin{column}{0.48\textwidth}
            \small
            \begin{itemize}
                \item Cliente humano (Obsidian) e cliente de software (agente de IA) consomem a mesma interface REST.
                \item O servidor encapsula a persistência: o cliente nunca acessa o armazenamento diretamente.
                \item O sistema é distribuído como uma imagem Docker que sobe o servidor e o banco em conjunto.
            \end{itemize}
        \end{column}
        \begin{column}{0.50\textwidth}
            \centering
            \begin{tikzpicture}[
                node distance=0.7cm and 0.6cm,
                client/.style={draw=azul-seg, thick, rounded corners, minimum width=2.2cm, minimum height=0.7cm, align=center, font=\small},
                core/.style={draw=azul-seg, very thick, fill=azul-seg!10, rounded corners, minimum width=3.2cm, minimum height=0.9cm, align=center, font=\small\bfseries},
                store/.style={draw=azul-seg, thick, rounded corners, minimum width=2.4cm, minimum height=0.7cm, align=center, font=\small},
                container/.style={draw=azul-seg!60, thick, dashed, rounded corners, inner sep=0.4cm},
                arrow/.style={-Stealth, thick, azul-seg}
            ]
                \node[client] (obs) {Plugin Obsidian};
                \node[client, right=1.0cm of obs] (ai) {Agente de IA};
                \node[core, below=of $(obs)!0.5!(ai)$] (api) {API REST (Go)};
                \node[store, below=of api] (db) {SQLite};

                \draw[arrow, <->] (obs) -- (api);

                \draw[arrow, <->] (api) -- (db);

                    \node[container, fit=(api) (db), label={[font=\scriptsize\itshape, azul-seg]below:Imagem Docker}] {};
                \end{scope}

        \end{column}
    \end{columns}
\end{frame}

\begin{frame}{Solução: decisões técnicas}
    \begin{itemize}\small
        \item \textbf{Servidor em Go}: alto desempenho e baixo consumo de recursos, mantendo o sistema leve para \textit{self-hosting} em hardware modesto.
        \item \textbf{SQLite como armazenamento}: zero configuração; conteúdo, anexos e metadados gravados em uma única transação; \textit{backup} reduzido a copiar um arquivo.
        \item \textbf{API REST}: contrato HTTP documentado que permite a qualquer cliente, humano ou agente de IA, integrar-se ao sistema.
        \item \textbf{Plugin do Obsidian em TypeScript}: cliente humano de referência; o servidor permanece aberto a outros clientes.
        \item \textbf{Distribuição via Docker}: imagem reprodutível que sobe servidor e banco em conjunto, reduzindo o atrito de instalação.
    \end{itemize}
\end{frame}

\section{Alternativas no ecossistema}
% -------------------------------------------------%

\begin{frame}{Alternativas no ecossistema}
    \begin{itemize}\small
        \item \textbf{Notion}: rico em \textit{features}, porém proprietário e com os dados na nuvem do fornecedor.
        \item \textbf{Obsidian com Sync}: cliente excelente, mas sincronização proprietária e paga.
        \item \textbf{Logseq self-hosted}: aberto e local, focado no cliente, sem servidor de fato.
        \item \textbf{Markupp}: servidor aberto, API REST como interface primária e integração pensada para agentes desde o projeto.
    \end{itemize}

    \vspace{0.5em}
    O Markupp não disputa um ranking: ocupa uma interseção rara do ecossistema, entre servidor aberto, \textit{self-hosting} real e abertura a agentes.
\end{frame}

\section{Feedback}
% -------------------------------------------------%

\begin{frame}{Feedback}
    O retorno até aqui é inicial e interno; o projeto mantém um canal aberto para feedback da comunidade:
    \begin{itemize}
        \item Repositório público com \textit{issues} e \textit{pull requests} abertos a qualquer pessoa, o espaço onde o feedback externo é recebido.
        \item \textit{Dogfooding}: o próprio time usa o Markupp como cliente Obsidian sobre o servidor.
        \item Validação de que a arquitetura sustenta o caso de uso real, com CRUD operacional de ponta a ponta entre plugin, API REST e SQLite.
        \item \textit{Release} v0.2.0 publicada como primeiro passo de exposição pública.
    \end{itemize}
\end{frame}
